# 多维评测系统

状态：**已实现基础闭环。** 核对日期：2026-07-24。

评测系统用于回答“当前版本有哪些可验证证据、哪些必须看片/试听、是否值得进入下一阶段”，而不是
用一个自动总分冒充导演、表演或审美判断。

## 三个检查点

| 检查点 | 适用阶段 | 维度与权重 |
|---|---|---|
| 剧本 / 分镜 | 付费画面前 | 剧本与情绪 30%、人物与声音身份 20%、分镜与镜头语言 30%、竖屏发行 20% |
| 样片 / 作监 | 关键帧、视频 take 与声音样片 | 剧本 12%、人物 14%、分镜 18%、画面 22%、配音 17%、连续性 17% |
| 成片 / 交付 | 合成和 picture lock 前 | 剧本 8%、人物 9%、分镜 13%、画面 17%、配音 14%、连续性 14%、平台 10%、交付 15% |

自动检查读取结构化剧本、资产锁定、分镜、镜头状态、候选/圈选文件、音频、视频、成片和 QC。
每项证据注明：

- `direct`：直接文件或机器可验证事实，覆盖率计 1；
- `proxy`：只能间接支持质量判断，覆盖率计 0.5；
- `missing`：没有证据，覆盖率计 0。

## 自动与人工不混淆

没有人工评分时，维度显示自动结构分，但报告不会因此声称审美或表演已经通过。人工看片/试听后，
可对当前检查点的维度给出 0–100 分并记录具体镜头、台词或声音证据。两类证据同时存在时：

```text
维度分 = 自动结构分 × 45% + 人工感知分 × 55%
```

总分按检查点维度权重汇总。判定规则：

- 总分低于 70：`revise`；
- 总分不低于 70，但人工覆盖低于 50% 或自动证据覆盖低于 60%：`needs_human_review`；
- 其余：`pass`。

这不是平台审核结论，也不是播放量预测。人工评分必须绑定证据，团队应按统一样片校准尺度。

## 不可变报告与过期判定

内容评测保存在：

```text
episodes/<EP-ID>/evaluations/<EVALUATION-ID>.yaml
```

报告创建时记录系列、剧本、分镜、资产、关卡、镜头和交付状态的 `inputHash`。任何相关输入变化后，
读取接口会返回 `stale: true`；历史文件仍保持不变，便于比较修订前后版本。

## LibTV / 本地管线同量表对比

候选池可包含 LibTV 已回收结果、本地已选关键帧、已选视频 take 和最终成片。Studio 一次选择
2–4 个候选（API 上限为 12 个），用 `amnTV-perceptual-v1` 量表人工评分：

| 产物 | 评分项 |
|---|---|
| 图片 | 人物一致性 30%、构图 25%、镜头语言 20%、干净度 25% |
| 视频 | 人物一致性 20%、构图 15%、镜头语言 20%、动作 20%、干净度 15%、配音表演 10% |

对比报告同时探测图片宽高，或视频宽高、时长、帧率和音轨；保留 provider、model、已知费用和人工
备注。费用缺失时必须显示“未知”，不能用估算值代替。报告保存在：

```text
episodes/<EP-ID>/evaluations/benchmarks/<BENCHMARK-ID>.yaml
```

dry-run 结果只验证候选发现、评分、排序和持久化，不代表 LibTV 或任何供应商的真实质量。

## API

```text
GET  /api/series/:series/episodes/:episode/evaluations
POST /api/series/:series/episodes/:episode/evaluations
GET  /api/series/:series/episodes/:episode/benchmarks/candidates
GET  /api/series/:series/episodes/:episode/benchmarks
POST /api/series/:series/episodes/:episode/benchmarks
```

请求由 Zod 校验；报告使用新 UUID 写入，不覆盖历史记录。Studio 的 10 阶段工作流只把评测作为
质量支路展示，最终成片批准仍由人工关卡③完成。
